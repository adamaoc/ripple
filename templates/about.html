About Ripple

A shared task system where humans plan and review work while coding agents implement it.

01

What it is

Ripple is a self-hosted project backlog and agent runner. Stories provide durable context that survives individual coding sessions. Humans decide what enters the queue and what gets closed; agents can discover stories through the API, implement them, and report progress without needing the workflow explained from scratch each time.

Backlog→Queued→In progress→In review→Done→Closed

Done always means merged. In review means a pull request is open and the ball is with you (or with an autonomous fix pass still running).

02

How to use it

  1. Create a project.Use the dashboard form or the API. Every story belongs to a project and may optionally belong to an epic.
  2. Shape a project backlog.Review story descriptions, set project autonomy and delivery options, select stories, and arrange the queue in execution order.
  3. Configure agents once.In Settings, bind the Implementer (Codex CLI) and Reviewer (Grok CLI or an OpenAI-compatible API). Path env vars always win over Settings.
  4. Start a run.The queue is frozen into a run. The implementer works each story, the reviewer inspects the PR, and the transcript records commands, reviews, failures, and summaries.
  5. Autonomous vs supervised.Autonomous projects implement, review, quality-gate, and merge without waiting. Supervised projects open a PR, post an agent review, then stop in In review. You act on review comments, merge when ready, or sync if you already merged on GitHub.
  6. Close when finished.Stories reach Done only after merge. A human can then close the story to archive it with its history.
03

Agent and tool requirements

The runner launches agents from each project's working directory. Defaults are local CLIs; the Reviewer may also be an HTTP API configured under Settings.

C

Codex CLI

Implementation and fixes

The default Implementer uses non-interactive codex exec with JSON event output so activity can appear in the live transcript.

codex --versionOverride discovery with RIPPLE_CODEX_BIN or Settings → Agents.
G

Grok CLI

Independent pull-request review

The default Reviewer returns approval or concrete review comments for the implementer to address. You can swap in an OpenAI-compatible API reviewer in Settings.

grok --versionOverride discovery with RIPPLE_GROK_BIN or Settings → Agents.
GH

GitHub CLI

Pull requests and merge workflow

The gh CLI creates, inspects, and merges pull requests. It must be authenticated for the repository.

gh auth statusOverride discovery with RIPPLE_GH_BIN.

Before a run: set the project working directory to its Git repository, use Verify setup in project settings, keep a clean working tree, and confirm the remote is accessible. API keys for reviewers stay local and are never written into transcripts.

04

Using the API with your own coding agent

The API is intentionally discoverable. Point an agent at the API root first; it links to the human-readable guide and the OpenAPI description.

GET /apiDiscovery and workflow rulesGET /api/docsBot-focused usage guideGET /api/openapi.yamlMachine-readable API contract

Give this instruction to a coding agent

You are working with Ripple at http://localhost:8080.

Start by requesting GET /api. Read /api/docs and /api/openapi.yaml before changing task data. Find the requested story by ID, read its details and event history, and treat its description as the source of truth for the work.

When you begin implementation, move the story to in_progress through the API. Keep your work scoped to that story, verify the result, then move it to done and summarize what changed. Do not move stories to queued or closed; those actions belong to the human workflow.

An agent can also create a new backlog story through POST /api/stories. Give it the project ID or project name, a clear title, and a Markdown description containing the goal, context, constraints, and completion criteria.
